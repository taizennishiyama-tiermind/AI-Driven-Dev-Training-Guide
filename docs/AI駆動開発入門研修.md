# AI駆動開発入門研修｜完全骨子

---

## 全体コンセプト

**一文要約：**
「AIに作ってもらう。ただし、うまく作ってもらうための材料は人が渡す。」

**研修の性格：**
- 開発の専門教育ではない
- 「AI時代の実装の進め方」を理解し、実務で自走できる状態に近づける研修
- 基本はClaude Codeが自律的に最適化する前提。人間は最小限の方向性・材料・制約を渡す
- 小難しくしない。ただし本番運用に耐える知識は入れる

**設計思想：**
1. まず作れるようにする（成功体験）
2. 次に品質を上げるコツを入れる（デザインシステム、スキル図、テスト）
3. 最後に実務運用の勘所を入れる（API、セキュリティ、LLM組み込み）

**利用ツール：**
- GitHub（変更履歴と共同開発の基盤）
- Claude Code（実装の主役）
- VS Code（人とAIの作業ハブ）
- Google AI Studio / Build機能（初期の発想整理・試作加速）
- Vercel（最短で公開する場所）
- Cloudflare Workers（本番寄りの拡張先）

---

## 到達目標

研修が終わった時点で、受講者が以下の状態になっていること。

1. AI駆動開発の考え方（人が材料を渡し、AIが自律的に組み上げる）を理解している
2. Claude Codeに「基本的に作ってもらう」感覚を持てている
3. 何をAIに渡せばいいか（願望・前提・制約・材料）が分かっている
4. `assets/`・`manifest.md`・`CLAUDE.md` の使い方が分かる
5. ローカルで確認しながらフィードバック→改善のループを回せる
6. GitHubで変更履歴を管理する意味と基本操作が分かる
7. テストの必要性と「どこまで固めるか」の判断軸を持てている
8. API・秘密情報の扱いで危険なことを避けられる
9. LLMを「生成」と「分類」で使い分ける発想を持てている

## 第1章｜AI駆動開発とは何か

### 1-1. 章の目的

AI駆動開発の考え方を、難しくなく掴ませる。「AIは補助ツール」という先入観を外す。

### 1-2. 伝えるコアメッセージ

> **「AIに作ってもらう。ただし、うまく作ってもらうための材料は人が渡す。」**

### 1-3. 具体的に伝える内容

**AI駆動開発の定義（シンプルに）：**
- AI駆動開発 ＝ 人が願望・前提・制約・材料を渡し、AIが自律的に組み上げる開発スタイル
- 用語をわかりやすく：
  - 願望 → 何を作りたいか
  - 前提 → 誰が使うか、どこで使うか、何のためか
  - 制約 → セキュリティ、人数、予算、環境、期限
  - 材料 → データ、参考資料、既存ファイル、デザインの方向性

**従来との違い：**
- 従来：人が設計→人が実装→AIは補完
- AI駆動：人が材料を渡す→AIが実装・整理・改善・テスト補助まで組み上げる
- Claude Codeは「補助ツール」ではなく「実装の主役」

**「設計しすぎない」という考え方：**
- 技術選定を細かく決めすぎると、AIの最適化余地を潰す
- 古い知識でライブラリを固定すると、最適でないものを指定してしまうリスク
- 基本はAIに任せる。こだわりたい部分だけ人が握る
- 知識がある人は、その知識でこだわる箇所を明確にする

### 1-4. 受講者の先入観を外すポイント

| よくある誤解 | 実際 |
|---|---|
| AIは補助的に使うもの | AIを主戦力として使い、材料設計を人がする |
| 全部自分で設計してからAIに渡す | 目的・前提・制約・材料だけ渡せばいい |
| コード補完みたいなもの | ゼロから組み上げまでやってくれる |
| 要件を短く正確に定義しないといけない | ラフでもいいから多く伝えた方がいい |

### 1-5. 講師が使うフレーズ例

- 「自分で全部書く前提ではなく、Claude Codeに作ってもらう前提でいいです」
- 「細かく指定しすぎると、むしろAIの良さが消えます」
- 「基本は任せる。必要なところだけ人が握る」

## 第2章｜最初に何を渡せばうまくいくか

### 2-1. 章の目的

AIに渡すべき情報を整理し、「入力の質」の重要性を理解させる。

### 2-2. コアメッセージ

> **「最初から綺麗に書かなくていい。まずはできるだけ多く伝える。」**

### 2-3. AIに渡す7つの材料

#### ① 願望（何を作りたいか）
- 何を作りたいのか
- どういう体験を作りたいのか
- 何ができると嬉しいのか
- 何を改善したいのか
- **綺麗に言語化しなくてよい。ラフでもいいので量を出す。**

#### ② 利用者情報（誰が使うか）
- 社内向きか、顧客向きか
- 一人か、チームか
- ITリテラシーは高いか低いか
- 同じ機能でも「社内管理者向け」「一般ユーザー向け」「高齢者向け」「子ども向け」でUI設計が根本から変わる
- **ペルソナ設計までは不要だが、最低限「誰が使うか」は明確にする**

#### ③ 利用規模（何人くらいか）
- 何人くらいが使うのか
- 同時アクセスはあるか
- 一時的な利用か、継続運用か
- 小規模プロトタイプか、本番前提か
- **これでテストの厚み、構成の堅さ、パフォーマンス要件が変わる**

#### ④ 機密性・セキュリティ要件
- 機密情報を扱うか
- 個人情報を扱うか
- APIキーや認証情報を扱うか
- 外部公開するか
- **機密性が高いなら最初に伝える。AIは「とりあえず動く構成」に寄りがちなので、ここは人が握る**

#### ⑤ 動作環境（どこで動かしたいか）
- ローカルだけか、Vercelに出すか、Cloudflare系か
- ブラウザアプリか、モバイル前提か
- 社内環境で閉じたいか
- **環境が整理されていれば、AIがそこに最適化して作ってくれる**

#### ⑥ こだわり（人が握るべき部分）
- 見た目の雰囲気
- ブランドトーン
- シンプルさ / 高級感 / スピード重視 / 保守性重視
- 使ってほしくない技術
- 必須機能
- 絶対に外してほしくない体験
- **ここだけは明確にする。それ以外はAIに任せる**

#### ⑦ 実データ・既存資料・参考情報
- CSV / Excel / 画像 / PDF / ドキュメント / API仕様書
- 参考スクショ / ロゴ / ブランドガイド
- **言葉で説明するより実物を渡した方が早い。AIにとっても精度が上がる**

### 2-4. 音声入力の活用

**なぜ音声入力が有効か：**
- 文章にしようとすると止まる人が多い
- 話すなら意外と出てくる：「こういう人が使って」「こういう課題があって」「こんな画面がほしくて」
- AI駆動開発では「ラフな多量入力」がとても強い

**講師が使うフレーズ例：**
- 「最初から綺麗に書かなくていいです。まずは音声入力で、やりたいことを一気に話してください」
- 「頭の中にあることをとにかく出す。整理はあとでいい」

### 2-5. ハンズオン案

1. 受講者が自分の作りたいものを音声入力で2〜3分話す
2. その出力を `manifest.md` にそのまま貼る
3. Claude Codeに渡して、何が生まれるか見る
4. → 「こんなに雑でも動くものが出るんだ」という体験

## 第3章｜材料は言葉だけでなく"ファイル"で渡す

### 3-1. 章の目的

`assets/` フォルダの考え方を定着させる。

### 3-2. コアメッセージ

> **「言葉だけで説明しなくていい。材料があるなら最初から渡す。」**

### 3-3. `assets/` フォルダの運用

**プロジェクトルートに `assets/` フォルダを作り、以下を入れる：**

| 種類 | 具体例 |
|---|---|
| 元データ | CSV / Excel / JSON |
| 参考ビジュアル | 参考画像 / スクリーンショット / ロゴ |
| ドキュメント | PDF資料 / API仕様メモ / 社内資料 |
| デザイン | デザインガイド / ブランド資料 / 参考UI画像 |
| 既存資産 | 既存ファイル / テンプレート / 文言集 |

**AIにとってのメリット：**
- 何を扱うのか判断しやすい
- どういう見た目が近いのか分かる
- どんなデータ構造か把握できる
- 何を前提に設計すべきか理解できる

### 3-4. 講師が使うフレーズ例

- 「データや資料があるなら、最初から `assets/` に入れておきましょう」
- 「言葉で説明するより早いです」

## 第4章｜`manifest.md` と `CLAUDE.md`

### 4-1. 章の目的

「何を作るか」の前提メモ（manifest.md）と「どう作るか」のAI向けルールファイル（CLAUDE.md）の違いを理解させる。

---

### 4-2. `manifest.md`（前提メモ）

**位置づけ：**
- 型に当てはめる書類ではない
- 作りたいものについての前提メモ
- 人間の頭の中にあることを外に出す場所
- AIに前提を渡す場所
- 途中でブレたときに戻る軸

**「要件定義書」ではない：**
- 「要件定義」という言葉で身構えさせない
- 綺麗に書く必要はない
- 音声入力で一気に話したものをそのまま入れてよい

**最低限入っているとよい項目：**
- 誰が使うか
- 何を実現したいか
- どこで使うか / どこで動かしたいか
- 何人くらいが使うか
- 機密情報を扱うか
- 今回のゴール
- こだわりたい点
- 使いたいデータ / assets に何があるか
- 補足の願望（あれば何でも）

**テンプレは配らない方針：**
- 型に当てはめることが目的ではない
- 「ちゃんと言語化する」ことが大事
- ただし最低限の観点リスト（上記）は配ってよい

**講師が使うフレーズ例：**
- 「綺麗に要件定義書を書く必要はありません」
- 「manifest.mdは、作りたいものについての前提メモです」

---

### 4-3. `CLAUDE.md`（AI向けルールファイル）

**位置づけ：**
- このプロジェクトで、Claude Codeにどう動いてほしいかを伝えるファイル
- AI駆動開発の品質を安定させるための「開発ルールの受け渡し」
- `manifest.md` が「何を作るか」なら、`CLAUDE.md` は「どういう方針で作るか」
- **こちらはテンプレをある程度渡してよい**

**なぜ必要か：**
- AIは毎回同じ方向で動くとは限らない
- 毎回のプロンプトだけで運用すると、実装方針・命名・UI雰囲気がブレる
- `CLAUDE.md` があると一貫性を持って働いてもらえる

**入れるとよい内容：**
1. プロジェクトの目的
2. 優先順位（速度 / 品質 / 保守性 / シンプルさ）
3. 想定ユーザー
4. デプロイ先
5. 使いたい技術 / 避けたい技術
6. UI/UXの方向性
7. セキュリティ注意点
8. データの扱い
9. テストの方針
10. GitHub運用（必要なら）
11. 実装時の優先ルール
12. 完了条件

**配布するたたき台サンプル：**

```markdown
# CLAUDE.md

## Project Goal
このプロジェクトは、○○を実現するWebアプリを作ることを目的とします。
まずはローカルで動く最小構成を作り、必要に応じて改善します。

## Priority
1. まず動くこと
2. シンプルでわかりやすいこと
3. 保守しやすいこと
4. 必要以上に複雑にしないこと

## Users
主な利用者は○○です。
複雑なUIや専門用語は避け、迷いにくい導線を優先してください。

## Environment
- まずはローカルで起動確認できる状態を作る
- 必要に応じてVercelへデプロイしやすい構成にする
- 環境変数が必要な場合は `.env.example` を用意する

## Implementation Policy
- まず最小構成で実装する
- 一度に過剰実装しない
- 必要な機能から順に追加する
- 読みやすい構成を優先する
- ディレクトリ構成はシンプルに保つ

## UI / UX
- 見た目はシンプル・クリーン・わかりやすく
- 迷いにくいUIを優先する
- ボタンや入力欄は明確にする
- まずは最低限の画面でよい
- 過度なアニメーションや装飾は不要

## Data / Assets
- `assets/` 配下にあるファイルを必要に応じて参照する
- データ形式を確認し、扱い方が不明な場合は仮定を明示する

## Security
- APIキーや秘密情報をフロントエンドに直書きしない
- 機密情報を扱う場合は、安全な構成を優先する
- 認証情報は環境変数で扱う

## Testing
- 実装後はローカルで起動確認できる状態にする
- 主な導線は最低限確認できるようにする
- 重要なロジックにはテストを追加する
- テストは壊れると困る箇所を優先する

## Git / Delivery
- 変更内容がまとまったらコミットしやすい単位に整理する
- 必要に応じてPR向けに変更概要をまとめる

## Completion Criteria
- ローカルで起動できる
- 主要機能が最低限動く
- UIが大きく破綻していない
- 次に直すべき点が明確になっている
```

## 第5章｜Claude Codeで実装する

### 5-1. 章の目的

AIと一緒に実装を進める流れを掴ませる。「見ながら育てる」感覚を体得させる。

### 5-2. コアメッセージ

> **「最初から完璧を狙わず、まず動くものを作る。見ながら直す。」**

### 5-3. 基本フロー

```
① 作りたいことをClaude Codeに伝える（manifest.md + 音声入力）
② Claude Codeが実装する
③ ローカルで動かす（localhost で確認）
④ 触ってみる → 違和感を見つける
⑤ フィードバックする（「ここが使いづらい」「文言を変えて」「色を変えて」）
⑥ Claude Codeが修正する
⑦ ③〜⑥を繰り返す
⑧ 十分になったら次のステップ（GitHub / デプロイ）へ
```

### 5-4. localhost で確認する意味

**なぜローカル確認が重要か：**
- AIの「できました」を鵜呑みにしない
- 実際に触ると必ず見つかる：
  - 動線が分かりづらい
  - 文言が不自然
  - 思った動きと違う
  - レイアウトが崩れている
  - データの取り回しが想定と違う
- 小さな違和感を早く見つけられる → 早く直せる

**講師が使うフレーズ例：**
- 「実装できたら、まず localhost で触ってください」
- 「見てから直せばいいんです。文章上で完璧を目指すと逆に遅くなります」

### 5-5. フィードバックのコツ

**伝え方の例：**
- ✅「このボタン、もう少し目立たせて」
- ✅「ヘッダーの文言を○○に変えて」
- ✅「この画面、スマホだと崩れてるから直して」
- ✅「この処理、エラーの時にメッセージが出ないから追加して」
- ❌「なんか違う」（→ どこがどう違うか伝える方がAIの修正精度が上がる）

**大事な感覚：**
- 最初の出来は粗くてよい
- 見てから直せばよい
- 動くものが出るとフィードバックしやすい
- AIは修正の反復に強い（何十回でも嫌がらない）

### 5-6. デザインの扱い方

**コアメッセージ：**
> **「デザインをゼロから考えなくていい。好きな方向があるなら渡す。なければ、用意された基準を使えばOK。」**

**進め方：**
1. 研修側でデザインシステムのMarkdownドキュメントを用意して渡す
2. 好みがあるなら、それをCLAUDE.mdに書くか、参考画像を `assets/` に入れる
3. 好みがないなら、渡されたデザインシステムをそのまま使う
4. 自分好みに整えたいなら、そのMarkdownを編集していく

**デザインシステムドキュメントに含まれるもの（例）：**
- カラーパレット
- タイポグラフィ（フォント、サイズ）
- スペーシング基準
- ボタン・フォーム・カードなどのコンポーネント方針
- トーン（クリーン / ポップ / ミニマル / etc.）

**講師が使うフレーズ例：**
- 「デザインで止まらないでください。まずはデフォルトで出してもらって、後から調整すればいいです」

### 5-7. クオリティを上げるコツ：スキル（Skills）の紹介

**スキルとは：**
- Claude Codeが特定の作業をうまく行うための「ベストプラクティス集」
- 例：iOS風UIを作る、フロントエンドを高品質にする、プレゼンを作る、etc.
- Claude Codeの `/mnt/skills/` に配置されている

**使い方のイメージ：**
- プロジェクトに合ったスキルをCLAUDE.mdで指定する
- 例：「UIはiOS風デザインシステムに従ってください」
- Claude Codeがそのスキルを参照して実装の質を上げる

**講師が使うフレーズ例：**
- 「スキルを指定するだけで、AIの出力の質がグッと上がります」
- 「いい感じのデザインにしたいなら、デザインスキルを指定するだけでOK」

### 5-8. よく使うプロンプト集（実装編）

受講者に配布するもの。そのままClaude Codeに投げられる：

```
■ 初回実装
「manifest.md と CLAUDE.md を読んで、最小構成で実装してください」

■ ローカル起動
「localhost でデプロイして確認できるようにしてください」

■ UI修正
「○○の部分を修正してください。具体的には△△のように変えたいです」

■ 機能追加
「○○の機能を追加してください。仕様は△△です」

■ エラー修正
「○○のエラーが出ています。修正してください」
（エラーメッセージをそのまま貼り付ける）

■ リファクタリング
「コードを整理してください。読みやすさを優先してください」
```

## 第6章｜GitHubと実務運用の流れ

### 6-1. 章の目的

AIが作ったコードを実務の管理下に置く考え方を伝える。

### 6-2. なぜGitHubが重要か

- AIが高速に変更を加えるほど、差分管理の重要性が増す
- 「何を変えたか」「戻せるか」「レビューできるか」の基盤
- AIに自由に触らせるからこそ、管理が必要

### 6-3. 伝える内容

- コードのバージョン管理
- 変更差分の確認
- コミット履歴の保持
- Pull Requestによるレビュー導線
- 「どこがどう変わったか」の可視化

### 6-4. よく使うプロンプト集（GitHub編）

受講者に配布するもの：

```
■ 新規プロジェクト開始
「新しいディレクトリを作成して開発を始めてください」

■ ローカル起動確認
「まずローカルで起動可能な状態にしてください」

■ コミット整理
「主要変更をコミットしやすい単位でまとめてください」

■ 変更要約
「変更内容を要約してください」

■ GitHub反映
「GitHubにプッシュしてください」

■ PR作成
「Pull Request用の説明文を作ってください」

■ PR提出
「GitHubにPull Requestを出してください」
```

## 第7章｜テストと品質

### 7-1. 章の目的

本番運用を前提にした最低限の品質感覚を持たせる。

### 7-2. コアメッセージ

> **「全部を完璧に守るより、壊れると困るところから守る。」**

### 7-3. テストの3層

#### 層1：手で触る確認（必須）
- `localhost` で実際に触る
- 主要導線が成立するか確認
- UIの違和感をチェック
- **これは毎回必ずやる**

#### 層2：重要ロジックのテスト
- 対象：計算、バリデーション、データ変換、条件分岐、分類
- 壊れると困るロジックには簡単でもテストを書く
- **テストもClaude Codeに作ってもらってよい**

#### 層3：主要フローの確認
- ログイン、登録、検索、保存、送信 etc.
- 全部を網羅する必要はない
- 「重要な導線」だけは確認できる状態にする

### 7-4. テストの厚みは前提条件で変わる

| 前提 | テストの厚み |
|---|---|
| 自分だけが使う小さな試作 | 手で触る確認だけでもOK |
| 部署内で使う簡易ツール | 重要ロジックのテストまで |
| 顧客が使う本番プロダクト | 主要フローまでしっかり |
| 個人情報を扱う業務システム | ガチガチに固める |

**判断軸：**
- 何人が使うか
- 機密情報を扱うか
- 失敗した時の影響
- どのくらい継続運用するか

### 7-5. よく使うプロンプト集（テスト編）

```
■ テスト作成
「重要なロジックにテストを追加してください」

■ テスト実行
「テストを実行して結果を教えてください」

■ 動作確認
「主要な導線が正しく動くか確認してください」
```

## 第8章｜APIと外部連携

### 8-1. 章の目的

実務で避けて通れない「つなぎ込み」の考え方を理解させる。

### 8-2. コアメッセージ

> **「AIが作ってくれても、何とどうつながるかは人が理解しておく。」**

### 8-3. APIの基本概念（技術講座にしない）

**APIとは：**
- アプリと外部サービスをつなぐ窓口
- 実務では外部サービスやデータと連携することが多い
- AIが実装してくれても、接続の考え方は人が理解しておくべき

**最低限理解すべき6項目：**
1. どことつながるのか（連携先）
2. 何を送るのか（リクエスト）
3. 何が返ってくるのか（レスポンス）
4. 認証が必要か（APIキー、トークン）
5. どこに秘密情報を置くべきか（環境変数）
6. フロントだけで完結させてよいか（セキュリティ判断）

### 8-4. 本番運用を意識した秘密情報の扱い

**必須メッセージ：**
> **「秘密情報はフロントエンドに直置きしない。」**

**フロントに置いてはいけないもの：**
- APIキー
- シークレット
- DB接続情報
- 外部サービスの認証情報

**なぜダメか（シンプルに）：**
- ブラウザから見えてしまう
- 流出リスクがある
- 悪用される → コスト増、不正利用

**どうすべきか：**
- サーバー側で持つ
- 環境変数で管理する（`.env` ファイル）
- フロントは必要なAPIだけ叩く
- 直接秘密を持たせない

**講師が使うフレーズ例：**
- 「見える場所に鍵を置かない。それだけ覚えてください」
- 「APIキーはフロントに置かない。これは鉄則です」

## 第9章（番外編）｜LLMをアプリに組み込むときの設計

### 9-1. 章の目的

LLMを「生成だけ」で捉えない、実務的な設計感を持たせる。
アプリ内にAI/LLM機能を入れたい場合のAPI設定・セキュア運用の設計を教える。

### 9-2. いつこの章が必要になるか

**通常のWebアプリ** → この章は不要
**以下の機能をアプリに入れたい場合** → この章が必要：
- チャット機能
- AI応答生成
- テキスト要約
- 画像生成
- 自動分類・判定

### 9-3. LLMの使い方は2種類ある

#### パターンA：生成に使う
- LLMが最終出力を作る
- 例：チャット返答、要約文、メール文面、提案文、画像生成、テキスト書き換え
- 最も「AIっぽい」見え方

#### パターンB：判定・分類に使う
- LLMが最終出力を作らず、入力を解釈する役割だけ担う
- 例：問い合わせ内容の分類、テキストのカテゴリ分け、緊急度判定、意図判定、ラベル付け
- 入門者には見落とされがちだが、実務では非常に強い

### 9-4. 「分類だけAIに任せる」設計

**基本フロー：**
```
ユーザーが自由入力（非構造データ）
  ↓
LLMで内容を分類（「不具合報告」「要望」「質問」「緊急対応」etc.）
  ↓
分類結果に応じて分岐
  ↓
定型メッセージを返す / 担当部署に振り分ける / 別画面に遷移させる
```

**メリット：**

| 観点 | 生成型 | 分類型 |
|---|---|---|
| コスト | 毎回生成で従量課金高め | 入出力が短く安め |
| 安定性 | 文面がぶれる、想定外あり | 出力が安定、テストしやすい |
| 運用性 | 制御しづらい | ルールベースで制御可能 |
| 監査 | AIが何を答えるか不明 | AIは仕分けだけ。最終アクションは定義済み |

**講師が使うフレーズ例：**
- 「AIは答えを作るためだけでなく、入力を理解して仕分けるためにも使えます」
- 「毎回フル生成するより、分類だけAIに任せた方が安くて安定する場合がある」

### 9-5. 使い分けの判断軸

| こういう場合 | こちらを使う |
|---|---|
| 柔らかい返答・自然な生成が必要 | 生成型 |
| 安定・低コスト・制御重視 | 分類型 |
| 会話UI、提案文、要約 | 生成型 |
| 問い合わせ振り分け、意図判定、ラベル付け | 分類型 |

### 9-6. LLM組み込み時のセキュア設計

**必須ルール：**
- APIキーをフロントに置かない（鉄則・再掲）
- サーバー側からLLMを呼ぶ
- 利用量を管理する（従量課金の暴走防止）
- ログ・監査を考える
- ユーザーデータをどう送るか決める
- 送らないで済む情報は送らない

**LLMを組み込むときに増えるリスク：**
- 従量課金
- 情報送信（外部APIにデータを送る）
- レスポンス遅延
- 出力の揺れ
- プロンプトインジェクション

**講師が使うフレーズ例：**
- 「LLMをアプリに入れる場合は、機能を作るだけでなく、コスト・安全性・運用まで含めて設計する必要があります」

- 細かすぎる設計書

## 避けるべき誤解（研修で先回りして潰す）

| 誤解 | 正しい理解 |
|---|---|
| AIに全部丸投げできる | 作るのは任せられるが、前提・材料・制約は人が渡す |
| 最初に全部細かく決めないといけない | 細かく決めすぎるとAIの最適化余地を潰す |

| AIが作ったからテストしなくてよい | 本番運用前提なら確認とテストは必要 |
| とりあえず秘密情報も入れてしまえばいい | APIキーや機密情報はフロントに置かない |
